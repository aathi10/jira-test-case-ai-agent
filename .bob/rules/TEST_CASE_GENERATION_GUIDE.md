# JIRA Test Case Generation — AI Prompt Guide

> This document captures the exact guiderails and prompt structure used by Bob (AI Agent)

---

## How the Test Cases Were Generated

The CSV was **not** produced by a script. It was generated by Bob (the AI agent) reasoning
directly over the live JIRA issue fetched via the **MCP Atlassian tool**
(`mcp__atlassian__jira_get_issue`).

### Pipeline

```
1. Fetch JIRA issue via MCP
        ↓
2. Read: summary, description ("Why? / What?"), acceptance criteria, comments,
         linked issues, components, labels
        ↓
2b. Read relevant repo files via GitHub API (requires GITHUB_TOKEN in .env)
    - Searches REPO_URL_1 / REPO_URL_2 for files matching the issue key + components
    - Fetches file snippets (first 2000 chars) from each matched file
    - Enriches scenarios: targeted Functional / Security / Regression test cases
      are generated from actual file paths and content found in the repo
    - If GITHUB_TOKEN is not set, this step is skipped gracefully
        ↓
3. Apply the prompt guiderails below
        ↓
4. Output structured test cases → CSV / Markdown / JSON
   (repo-aware scenarios added under section k of ai-analyzer.ts)
```

### Required env vars

```ini
JIRA_USERNAME=your@email.com
JIRA_API_TOKEN=your_jira_token
GITHUB_TOKEN=ghp_your_github_token   # needed for Step 2b repo code reading
REPO_URL_1=https://github.com/ibm-webmethods/hybrid-ipaas-ui
REPO_URL_2=https://github.com/ibm-webmethods/hybrid-ipaas-tms
```

## The Master Prompt (copy this into any chat with Bob)

````
You are a senior QA engineer. I will give you a JIRA issue.
Read the full description, acceptance criteria, comments, replies and linked issues.
Then generate COMPREHENSIVE manual test cases following these guiderails:

═══════════════════════════════════════════════════════════
GUIDERAILS
═══════════════════════════════════════════════════════════

1. COVERAGE — generate test cases across ALL of these categories
   (only skip a category if it is genuinely not applicable):

   ✅ Functional      — happy-path & core behaviour
   ✅ Negative        — invalid input, error codes, boundary failures
   ✅ Integration     — API calls, data flow between systems
   ✅ Performance     — response time, load, concurrency
   ✅ Security        — auth, RBAC, HTTPS, data privacy
   ✅ Accessibility   — keyboard nav, screen readers, ARIA
   ✅ Documentation   — TODO comments, code comments, README accuracy
   ✅ Regression      — existing features must not break

2. STRUCTURE — every test case must contain:
   - ID:             <ISSUE_KEY>-TC-NNN  (NNN = zero-padded 3-digit number)
   - Title:          Action verb + subject  (e.g. "Verify … ", "Validate … ")
   - Priority:       Critical | High | Medium | Low
   - Type:           one of the categories above
   - Preconditions:  pipe-separated list of setup requirements
   - Test Steps:     numbered — "Action -> Expected Result" per step
   - Expected Result: one clear sentence summary
   - Test Data:      specific data values or mocks needed
   - Notes:          related issue keys, gotchas, performance targets

3. SCENARIO EXTRACTION — derive scenarios from:
   a) Every requirement listed in the "What?" section
   b) Every side-effect or edge case in the "Why?" section
   c) Every linked issue (look at their summaries)
   d) The fallback / error path for EVERY happy path
   e) Cross-browser, cross-session, cross-user isolation
   f) Feature flags / toggles (on AND off)
   g) Migration paths (old format → new format)
   h) Data cleanup steps (verify cleanup actually happened)

4. NAMING CONVENTION
   - Functional:    TC-001, TC-002 …
   - Negative:      TC-014, TC-015 … (continue the main sequence)
   - Performance:   TC-020, TC-024 …
   - Accessibility: TC-025, TC-026 …
   - Security:      TC-027 …
   - Documentation: TC-022 …
   (all in one flat numbered sequence, grouped by category in the output)

5. PRIORITY RULES
   - High:   any scenario that directly tests the core acceptance criteria
   - High:   any error-handling scenario involving data loss risk
   - Medium: edge cases, concurrent access, partial updates
   - Low:    documentation checks, future enhancements, audit trails

6. TEST STEP QUALITY
   - Step 1 is always navigation / setup
   - Each step has ONE action and ONE expected result
   - Use concrete actions: "Click X", "Navigate to Y", "Set Z to value"
   - Expected results are observable: "API returns 200", "Section collapses"
   - Minimum 4 steps per test case; complex flows may have 7–8

7. NOTES FIELD must include:
   - Any related JIRA issue keys (e.g. "Related to SCI-17508")
   - Performance targets where applicable (e.g. "< 500ms", "P95 < 3s")
   - Gotchas or special setup instructions

8. OUTPUT FORMAT
   CSV with these exact columns (quoted, comma-separated):
   Test Case ID, Title, Priority, Type, Preconditions,
   Test Steps, Expected Result, Test Data, Notes

═══════════════════════════════════════════════════════════
ISSUE TO PROCESS
═══════════════════════════════════════════════════════════

Fetch JIRA issue: <ISSUE_KEY>

Generate all applicable test cases and save to:
  test-cases/<ISSUE_KEY>-test-cases.csv   (CSV)
````

## Shorter "Quick" Prompt

For a fast generation with sensible defaults:

```
Fetch JIRA issue <ISSUE_KEY> via MCP and generate comprehensive manual
test cases. Cover: Functional, Negative, Integration, Performance,
Security, Accessibility. For each test case include: ID, Title, Priority,
Type, Preconditions, numbered Test Steps with expected results, Expected
Result, Test Data, Notes. Save as CSV and Markdown in test-cases/.
```

## Category-Specific Prompt Additions

Add any of these lines to the master prompt to get deeper coverage:

| Scenario | Add to prompt |
|---|---|
| Feature flag testing | `"For every feature (on/off), generate test cases for both flag states."` |
| Migration testing | `"Generate test cases for: data in old format only, new format only, both, neither, partial migration."` |
| Cross-browser | `"Generate test cases covering Chrome, Firefox, Safari, and private/incognito mode."` |
| API error codes | `"Generate negative test cases for HTTP 400, 401, 403, 404, 500, and network timeout."` |
| Data isolation | `"Generate test cases verifying that User A's data is never visible to User B."` |
| Rollback | `"Generate test cases for rollback scenarios: disable feature flag, revert DB, restore old config."` |
| Accessibility | `"Generate test cases for: keyboard-only navigation, screen reader announcements (JAWS/NVDA/VoiceOver), WCAG 2.1 AA compliance."` |

## What Bob Does Internally (the guiderails in code)

The generation follows this decision tree internally:

```
READ issue description
  └─ a) Extract "What?" → derive happy-path Functional scenarios
  └─ b) Extract "Why?" → derive cross-browser, incognito, private-window scenarios
  └─ c) Extract fallback chain → derive Negative + Error-handling scenarios
  └─ d) Extract feature flag → derive dual-mode (on/off) scenarios
  └─ e) Extract migration → derive: pre-migration, during, post, cleanup scenarios
  └─ f) Extract linked issues → derive Regression scenarios per linked ticket
  └─ g) Extract performance hints → derive Performance scenarios
  └─ h) Extract security hints → derive Security scenarios
  └─ i) Extract accessibility hints → derive Accessibility scenarios
  └─ j) Extract TODO/docs hints → derive Documentation scenarios
  └─ k) READ REPO FILES (NEW) → derive targeted code-aware scenarios:
         - Every matched file → Regression scenario
         - Files with route/component/handler → Functional scenario
         - Files with auth/role/token → Security scenario
         - Files with TODO/FIXME/deprecated → Documentation scenario
         Preconditions include: repo name + specific file paths found

FOR EACH scenario:
  → Determine category (Functional / Negative / Integration / Performance / Security / Accessibility / Documentation / Regression)
  → Assign ID (sequential, category-sorted in output)
  → Assign Priority (High if core AC, Medium if edge, Low if docs/audit)
  → Write 4–8 steps: setup → action → verify → check side-effects
  → Write expected result (single observable sentence)
  → Write test data (concrete values, mock configs, user accounts)
  → Write notes (related tickets, targets, gotchas, repo file paths)
```

## Example: Asking Bob for SCI-17066

This is exactly what was asked (paraphrased from session history):

```
Fetch JIRA issue SCI-17066 and generate comprehensive manual test cases.
The issue is about migrating IWHI platform user preferences from localStorage
to a personalization store. Cover all scenarios including:
- Happy path (preferences load from store)
- Migration (localStorage → store with cleanup)
- Fallback chain (store empty → localStorage → defaults)
- Feature flag (on/off, rollback)
- Cross-browser and incognito
- API error handling (500, timeout, 401)
- Concurrent sessions
- User isolation
- Performance benchmarks
- Security (HTTPS, auth, data privacy)
- Accessibility (keyboard, screen reader)
- Code quality (TODO comments)
Export as CSV to test-cases/SCI-17066-test-cases.csv
```

**Result:** 35 test cases across 7 categories, saved to CSV.

## Reuse Checklist

Before generating test cases for a new issue, verify:

- [ ] JIRA issue key is correct
- [ ] Issue has a description (not just a title)
- [ ] You know the output format you want (CSV / Markdown / JSON)
- [ ] You've noted any related issue keys to include in regression coverage
- [ ] You've decided whether to include Performance and Accessibility (adds ~5 test cases each)
